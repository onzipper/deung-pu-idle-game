# `render/` ‚Äî PixiJS rendering layer

Reads `GameState` from the engine and draws it with PixiJS (WebGL). **One-way:** render reads engine state, never mutates it.

Chosen over raw Canvas 2D because Pixi gives us particles/filters/glow for free and sidesteps two POC rendering bugs entirely:

- `IndexSizeError` from `ctx.arc()` with a negative radius (the `shockwave()` ring bug). Pixi doesn't call `arc` directly; still, any radius fed to a Pixi Graphic must be clamped `Math.max(0, r)`.
- `createRadialGradient` + `addColorStop` crashing when a CSS var resolved empty. Use Pixi filters instead of manual gradients.

Runs outside React. The Pixi `Application` is created once (client-only, `useEffect`) and driven by the engine loop ‚Äî it must **not** live in React state or re-render per frame.

## Public API (`GameRenderer.ts`)

```ts
import { GameRenderer } from "@/render/GameRenderer";

const renderer = new GameRenderer();
await renderer.create(canvasParent);       // once, client-only (e.g. inside a useEffect)
renderer.draw(state, frameEvents);         // every rAF; state is the engine's GameState,
                                            // frameEvents is this frame's sub-steps'
                                            // state.events concatenated by the caller
                                            // (see GameClient.tsx) ‚Äî optional/defaulted
                                            // to [] for backward compatibility.
renderer.destroy();                        // on unmount
```

- `create(canvasParent: HTMLElement): Promise<void>` ‚Äî async Pixi v8 `Application.init()`, appends `app.canvas` to `canvasParent`, builds the layer stack, and starts a `ResizeObserver` on `canvasParent` (Pixi's own `resizeTo` only reacts to `window` resizes, not layout-driven container resizes, so this class drives `renderer.resize()` itself). Idempotent ‚Äî calling `create()` again tears down any previous instance first, and `destroy()` is safe to call before `create()` ever resolved. This covers React StrictMode's dev-mode mount/unmount/mount.
- `draw(state: GameState, frameEvents: GameEvent[] = []): void` ‚Äî one-way read of engine state into Pixi display objects, then hands `frameEvents` to the `fx` layer's `FxController` (hit flashes, damage numbers, kill pops, screenshake, skill/boss VFX ‚Äî see below) and advances every fx timer by a real-wall-clock `dt` computed from `performance.now()` deltas (NOT the sub-step count, so a 2x/3x speed multiplier never fast-forwards the juice itself). Must be called by the integration layer's rAF loop (this class does not run its own ticker against engine state). Never mutates `state` or the events it's handed.
- `destroy(): void` ‚Äî full teardown: disconnects the resize observer, clears all entity pools, destroys the fx controller, the boss view/overlay, and the Pixi `Application` (`removeView: true` + children/texture cleanup).

### Scene layers (children of a `world` container, in this z-order)

`background` (static sky/ground/grid, drawn once) -> `entities` (heroes, enemies, boss) -> `projectiles` -> `fx` (M4 juice: damage numbers, particle bursts, expanding rings, full-arena flash ‚Äî see `fx/` below; hit-flash itself tints entity views directly in `entities`, not this layer) -> `overlay` (screen-anchored arena readouts; currently just the boss HP bar + enrage label, POC-faithful ‚Äî does not duplicate the React HUD).

`world` is scaled+letterboxed from a fixed logical coordinate space (`src/render/layout.ts`: `WORLD_WIDTH=900`, `WORLD_HEIGHT=300`, `GROUND_Y = CONFIG.layout.groundY`) to whatever pixel size the canvas actually is, so every view module draws in engine coordinates directly (no manual scaling math per shape) and resizing the container never rebuilds the scene.

### The two POC bug rules, enforced here

- Every radius/size passed to a Pixi `Graphics` call is wrapped in `safeRadius()` (`theme.ts`, `Math.max(0, r)`) ‚Äî see call sites in `hpBar.ts`, `enemyView.ts`, `bossView.ts`, `projectileView.ts`, `GameRenderer.ts`'s background/overlay bars.
- No hand-built canvas gradients anywhere; glow/soft edges (projectile halo, boss telegraph ring) are done with layered `alpha` on plain `Graphics` fills/strokes. Pixi filters (`pixi.js` built-ins, e.g. `ColorMatrixFilter`/`BlurFilter`) are the option to reach for once real bloom/glow is wanted ‚Äî not manual `createRadialGradient`.

## `fx/` ‚Äî M4 juice (event-driven, real-time, pooled)

`src/render/fx/FxController.ts` owns the `fx` layer's children and is the one-way consumer of a frame's collected `GameEvent[]` (see `engine/state/events.ts` and the collection contract in `GameClient.tsx`). Rule of thumb used throughout: **continuous/persistent** visuals (boss enrage aura, telegraph ring closing in) read `GameState` directly every frame in the relevant view module, same as HP bars; **edge-triggered, transient** juice (numbers, flashes, pops, shake) is driven from `consumeEvents()` here. All fx state is render-only and never touches `GameState`.

- `particles.ts` (`ParticlePool`) ‚Äî one shared, fixed-size ring-buffer pool of `Graphics` dots backing every burst effect (kill pops, skill/meteor impacts, boss-defeated burst + gold shower, wave-spawn poof, hero-revive sparkle, zone/map-unlock sparkle). `burst()`/`shower()` are thin spawn helpers over it. Radii always `safeRadius()`-clamped.
- `floatingText.ts` (`FloatingTextPool`) ‚Äî pooled, ring-buffer `Text` labels that rise + fade over a real-time duration. Two instances: damage numbers (cap 40, per spec) and a separate small "event text" pool (cap 16; kill/boss gold so it can't evict an in-flight damage number).
- `hitFlash.ts` (`HitFlashController`) ‚Äî brief "flash to white" on the hit target's own view container via a `ColorMatrixFilter` (map-to-white matrix + the filter's own `alpha` driven 1 -> 0 as the mix uniform); filters are attached only while flashing and returned to a free-list after, so idle entities pay zero extra render cost. No hand-built gradients.
- `rings.ts` (`RingPool`) ‚Äî pooled expanding/fading stroked-circle rings (swordsman spin cast, boss-telegraph intensify pulse, boss-slam-land shockwave). This is exactly the effect class the POC's negative-radius crash came from; every radius here runs through `safeRadius()`.
- `screenShake.ts` (`ScreenShake`) ‚Äî exponential-decay amplitude + rotating direction; `GameRenderer` composes its `offset` ON TOP of the letterbox `baseTransform` every `draw()` (`applyWorldTransform()`), never overwriting the resize math. Mild on `heroDown`, strong on `bossSlamLand`; a retrigger takes the max amplitude, not a sum.
- `arenaFlash.ts` (`ArenaFlash`) ‚Äî single reusable full-bleed rect for boss-enraged / boss-defeated / stage-advanced beats; peak-alpha kept subtle (~0.2-0.3) by design ‚Äî no strobing.
- `pixelWeaponFx.ts` (`createPixelWeaponFx`, M9 pixel-fx weapon port, `render/fx/` since this pass ‚Äî was `/lab` experiment ‚ë¶ support) + `refineFxRecipes.ts` (`resolveRefineFxRecipe`, was `/lab` experiment ‚ëß) ‚Äî the LIVE weapon fx: a PURE rarity+refine(+legendary)‚Üílayers recipe resolver feeding a pooled pixel-particle sim (square texels, flat-alpha palette-step fades, normal blend only ‚Äî footgun 10). One instance per hero slot, created LAZILY on first non-null recipe (mobile GPU budget), pool size 140 in-game (340 default in `/lab`). Runs `setStepFps(60)` in-game (smooth, code-drawn vectors) ‚Äî the module's stepped-12fps mode is for pixel-art SPRITE hosts only. Replaces the old `gearAura.ts` (tier-6/epic flame aura), the weapon half of `refinePrestige.ts` (+9/+10 crackle/beat), and `legendaryFx.ts` (per-class idle signature + swing trail) ‚Äî ALL retired; legendary weapons now ride the SAME recipe resolver (`resolveRefineFxRecipe(rarity, awakenLevel, true)`). Continuous, driven from `FxController.updateWeaponFx()` every frame (never an event) via `heroView.ts`'s `getWeaponAnchorPos()`/`isHeroAttackSwinging()` read-only hooks.
- `gearSparkle.ts` (`GearSparklePool`, M7) ‚Äî the tier-5+ armor looping sparkle/glint: a fixed cap of `MAX_SLOTS * GLINTS_PER_SLOT` (6 √ó 3 = 18, `MAX_SLOTS` mirrors `theme.ts`'s `MAX_PARTY_SIZE`) tiny pooled dots, twinkling in/out (not a constant glow) around the chest anchor (`getArmorAnchorPos()`). Same continuous-per-frame convention as `pixelWeaponFx.ts`. The armor half of `refinePrestige.ts`'s +9/+10 crackle/beat ladder still rides this piece's anchor (weapon half retired ‚Äî see above).
- `FxController.ts` ‚Äî wires the above to specific `GameEvent` types and exposes `consumeEvents(events, state)`, `update(dt)` (real seconds, never sub-step count ‚Äî so 2x/3x speed never fast-forwards the juice), and `shakeOffset`. Also owns `updateGearFx()` (continuous per-frame weapon-fx/armor-sparkle activation per hero slot, delegating the weapon half to `updateWeaponFx()`) and the `itemDrop` event handler (`onItemDrop()` ‚Äî a small rarity-tinted ring + burst pop at the drop's ground position, capped/subtle for common/rare, bigger for epic; knobs `ITEM_DROP_*` near the top of the file).
- `commandMarkers.ts` (M7.8 "Manual Play", `TargetLockReticle`) ‚Äî the tap-to-move/tap-to-attack feedback: `moveOrdered`/`targetLocked` (event-driven, via the shared `RingPool` ‚Äî a ground click-marker ripple + a lock-on pulse, `FxController.onMoveOrdered()`/`onTargetLocked()`) plus the PERSISTENT target-lock reticle itself, which is a CONTINUOUS per-frame read of `state.heroes[0].command` (same convention as `pixelWeaponFx.ts`, see `FxController.updateTargetLock()`) that eases its own alpha toward 0 when the command clears/completes instead of a hard cut. `GameRenderer.hitTestPointer()` (screen‚Üíworld tap hit-test against the live enemy list, monsters win over ground) is the one-way query `GameClient.tsx`'s canvas click/pointermove handlers use to turn a tap into a `moveTo`/`attackTarget` store intent ‚Äî render itself never dispatches intents.

`draw()` intentionally does not diff HP between frames to detect "just got hit" ‚Äî the `hit` event carries that instead, which is the whole reason the event buffer exists.

**M6 "World & Town" navigation beats** (all in `FxController.ts`): `zoneEntered` (a farm/town arrival, not a boss room) plays a soft full-width "whoosh" (a few faint horizontal streaks + the palette's gentlest camera-punch, `zoneWhoosh`) riding on top of `Environment`'s own biome crossfade ‚Äî zone display names are locale text, so naming the zone stays a UI-layer toast's job, never hardcoded here. `bossRoomEntered` gets its own dedicated, weightier entrance beat (shake + a subtle warm flash + a ring CONTRACTING inward, reading as "the gate shutting behind you") since the room itself is already visually distinct for the whole fight via its dedicated biome + `environment/bossArena.ts`'s framing. `zoneUnlocked`/`mapUnlocked` get a small/bigger congratulatory sparkle at the hero's position. **Cleanup**: the old `bossRetreat` event/beat (team wiped, boss backs off in place) is gone ‚Äî a wipe now walks home to town (`world.respawnToTown`); its dormant `fx`/audio handlers were removed rather than left dead, and `heroDown` was extended with a one-shot muted/desaturated flash (only on a full wipe) as the "somber" replacement beat, paired with a delayed second downward sweep in `audio/sfxMap.ts` (`playHeroWalkHome`).

### Art direction (binding for hero/enemy animation tasks too)

- **Palette philosophy:** biome scenery stays desaturated and mid-to-low value (dusk/twilight ranges), deliberately close in luminance to the old flat `arenaSky`/`arenaGround`. Combat entities (`HERO_COLORS`, `ENEMY_COLORS`, fx accents) stay saturated jewel tones. That contrast is the whole readability strategy under chaos ‚Äî scenery sets mood, entities read as "the important moving things." Hero/enemy sprite work should keep saturation high and reserve any biome-tinting to a *thin* rim-light/shadow accent in the current biome's `far.color`/`ground.accent`, never a wash over the whole sprite.
- **Silhouette shape language escalates with biome order:** rolling curves (meadow) -> tall jagged spikes (forest) -> sharp angular rock (cave) -> aggressive glowing-rim peaks (volcanic) -> sharp crystalline peaks (frost). This mirrors rising stage danger. Hero idle/march poses should read as grounded and rounded early (echoing meadow's calm), gaining sharper/faster silhouette breaks (weapon glints, more angular windups) as a run progresses; enemy silhouettes should get progressively spikier/more aggressive in the same arc, so a screenshot of any biome instantly signals "how far into the run" without reading numbers.
- **Motion grammar:** everything ambient (clouds, motes, leaves, snow) moves at a slow, constant, real-time pace ‚Äî calm by design. Only the far/near parallax "world travel" layers speed up/slow down with phase. Hero marching animation should borrow this same distinction: a steady baseline gait (like the parallax) that visibly quickens the instant `phase === "battle"` engages, and eases during `boss` standoffs ‚Äî keep hero locomotion tempo legible against the world's own tempo change rather than constant-speed.
- **No new gradients, ever:** every "soft" visual in this system (sky, horizon glow, glow rims, crystal glints) is layered flat-alpha rects/shapes or `tint`, per the POC-bug rule. Task 2/3 should keep reusing this vocabulary (layered alpha, not filters/gradients) for hit-glow, aura, or charge-up effects unless a Pixi filter is deliberately reached for.
- **Silhouette material vocabulary (PROCEDURAL V2, task 86d3k2nj3):** `theme.ts` adds three shared accents on top of the existing per-class/kind body hues ‚Äî `HERO_COLORS[cls].shade` (a darker in-family tone for hoods/robes/armor recesses ‚Äî the "2-3 flat tones per part" the shapes are built from), `PALETTE.steel` (neutral metal for blades/crossguards/arrowheads/staff bands, so armament reads as one material regardless of class), and `PALETTE.outline` (a thin near-navy stroke, not pure black, so armor/silhouette edges pop off scenery without turning into hard cutouts). Enemy kind personality (armor plates, angry eyes, hoods) reuses the SAME kind-coded body hue at layered black/white alpha rather than adding new per-kind palette entries ‚Äî `ENEMY_COLORS` stays "colors are kind-coded, one hex per kind." **Known footgun:** `Graphics.arc(...).fill(...)` (as opposed to `.stroke(...)`) can collapse toward the path's stale pen position instead of the arc's own coordinates ‚Äî build any filled curved cap from sampled points via `poly()` instead (see `heroView.ts`'s `arcFanPoints()`).

## World-depth seam & the engine-`y` cutover (R4)

`worldDepth/worldFxContext.ts` is THE shared pure seam every depth consumer (GameRenderer entity/hit-test placement, `GhostLayer`, `FxController` fx lift) resolves ground line + depth row through ‚Äî the depth/terrain flags flatten the whole world. Two contract notes:

- **Depth is engine-owned, single path (R4 Wave C0).** `depthOf` reads the engine's `entity.planeY` (Hero/Enemy carry it; ghosts pass engine-exported `scatterPlaneY(cid)`; town NPCs do NOT pass `scatterPlaneY` in R4 ‚Äî they stay on the static `DEPTH_NEUTRAL` path below for all of R4; `FxController` feeds the entity's stamped `planeY` for damage-number lift) and inverts it with `planeToDepth` ‚Äî the inverse is bit-exact (band width 64 = 2^n), so `depthOffsetY`/`depthScale`/`depthZIndex` reproduce the pre-cutover values EXACTLY. There is NO second path: the Wave-B `worldDepthFromEngineY` flag, its OFF branch, and the render-side depth-ASSIGNMENT (`depthAssign` heroDepth/enemyDepth/ghostDepth + HERO_* row constants) are all retired. `depthAssign` keeps its render-side `hashUnit` ONLY for the existing terrain-preset / weather-window consumers (`terrainZone.ts`, `weatherSchedule.ts`); the seam's defensive fallback for an actor with no engine `planeY` is the ENGINE's `hashUnit(id)`, matching the engine-owned plane scatter. Render imports the plane helpers + `hashUnit` from `@/engine` ‚Äî render‚Üíengine is the allowed import direction.
- **Not everything is depth-scattered ‚Äî and static actors stay static for all of R4.** Stage boss / world boss / town NPCs are placed via `placeStaticActor` at `DEPTH_NEUTRAL` (terrain lift only, fixed zIndex ‚Äî drawn frontmost/behind by explicit sort keys, not the depth band). Their engine `planeY` (the near row) is intentionally NOT consumed; feeding it through the band would move them ~40px and break the frontmost read. This static path is owner-confirmed to hold for the whole R4 arc (no y movement for these actors); revisit at R5 boss-telegraph work. Party heroes are fanned into the band at cohort build (`buildCohortState` stamps `heroPlaneY(cls, cohortIndex, size)`), so render reads the fan straight off engine state rather than re-deriving it.
- **Hero `planeY` is now LIVE (R4 Wave C1) ‚Äî kill-time fx need no event changes.** The engine steers a hero's `planeY` toward its engagement lane (see `docs/context/engine.md`). Render already tracks this correctly: `FxController.heroLift`/`hitY` look the hero up by id in `state.heroes` and read its LIVE `planeY` ‚Äî heroes are never removed from state, so hero-anchored fx (death wisp, revive pillar, damage numbers) follow the moved hero automatically. Enemy kill-time fx keep resolving the seam's spawn-hash fallback (`hashUnit(id)` when the dead enemy is gone from state) ‚Äî EXACT for static enemies since enemies never move in C1. Revisit only if C2/R5 makes enemies move (a moved enemy's kill-time fx would then stick to its spawn row and need `planeY` on its `kill` event).
